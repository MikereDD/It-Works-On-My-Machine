# 📝 v5.4.1 — Supported Video Sources Only

> Stop feeding the pipeline garbage.

---

## 🔥 Problem

The bot previously attempted to process **any detected URL**:

* Articles
* Tracking links
* Non-video pages
* Unsupported platforms

Result:

* yt-dlp failures
* Wasted processing time
* Noisy logs
* Unnecessary queue entries

---

## 🧠 Root Cause

URL detection was **too permissive**:

```text id="j3k3b9"
ANY URL → pipeline → yt-dlp → fail
```

There was no validation layer to ensure the URL was actually a **supported video source**.

---

## ✅ Fix

### ✔ Introduced Supported Domain Filter

```python id="p2y0fq"
SUPPORTED_VIDEO_DOMAINS = (
    "youtube.com",
    "youtu.be",
    "m.youtube.com",
    "instagram.com",
)
```

---

### ✔ New validation helper

```python id="q7n3z1"
def is_supported_video_url(url: str) -> bool:
```

Handles:

* domain normalization
* mobile → standard domain
* subdomain matching

---

### ✔ Early pipeline rejection

```python id="t2m9hz"
if not is_supported_video_url(url):
    return
```

👉 Happens **before queueing**

---

### ✔ Command-level protection

Applies to:

* `/dl`
* `/audio`
* `/clip`
* `/ui`

```python id="z8d4xp"
await update.message.reply_text("❌ Unsupported video source.")
```

---

### ✔ Watch folder filtering

```python id="r4n7vw"
if url and is_supported_video_url(url):
```

Only valid video links are processed.

---

### ✔ Hard safety layer

Also enforced inside:

* `get_media_info()`
* `download_media()`

```python id="m5x8dp"
raise RuntimeError("Unsupported video source.")
```

---

### ✔ Status visibility

```text id="c1n2vh"
/status
```

Now shows:

* supported video domains
