# Raziel v5.4.7

## 🛡️ Reply Protection Stabilization Release

v5.4.7 finalizes the reply-loop protection system introduced during the v5.4.5 and v5.4.6 fixes.

This release focuses on stability, behavior hardening, and cleanup of Telegram reply edge cases that could accidentally re-trigger downloads from Raziel’s own uploaded media.

---

## 🧠 Problem

Telegram reply payloads may contain nested metadata from the original replied-to message.

In some situations, replying to one of Raziel’s uploaded videos with simple text like:

```text
test
```

could unintentionally expose the original media caption URL to the fallback scanner.

Result:

```text
reply → repost loop
```

---

## ✅ Fixes

### Strict Reply Guard

Replies are now ignored unless the NEW reply message itself contains a fresh/direct URL.

Allowed:

```text
reply + new youtube link
reply + new instagram link
```

Ignored:

```text
reply + text
reply + emoji
reply + sticker
reply + reaction-style replies
```

---

### Hardened Payload Filtering

The raw Telegram fallback scanner now sanitizes:

```python
reply_to_message
```

before URL extraction occurs.

This prevents nested replied-to captions and metadata from being rediscovered during fallback scanning.

---

## 🎯 Result

Raziel no longer reposts uploaded videos when users casually reply to uploads.

Behavior is now:

* cleaner
* safer
* predictable
* stable in active group chats

---

## ⚙️ Internal

* Finalized reply-loop prevention behavior
* Hardened reply payload filtering
* Stable repost prevention release
* README + notes cleanup
* Version bumped to `v5.4.7`

---

## 📌 Philosophy

Automation should feel invisible.

The best behavior is often the behavior users never notice because it simply works correctly.
